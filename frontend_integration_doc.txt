MedicalAibot Backend API Integration Guide
========================================

Overview
--------
This backend exposes a FastAPI service for medical AI interactions. It connects to PostgreSQL, uses Groq with the gemma2-9b-it model, and runs a basic medical workflow on each chat request.

Base URL
--------
http://127.0.0.1:8000

Start the server
----------------
Run this from the project folder:
python -m uvicorn app:app --host 0.0.0.0 --port 8000

Environment variables
---------------------
Create a .env file with the following values:
DATABASE_URL=your_postgres_connection_string
GROQ_API_KEY=your_groq_api_key

Important notes:
- The backend reads the environment file automatically when the server starts.
- If the database or Groq key is missing, the corresponding endpoints will return an error response.

Available endpoints
-------------------
1) GET /
Description:
Returns a basic confirmation that the backend service is running.

Response example:
{
  "message": "MedicalAibot backend is running"
}

2) GET /health
Description:
Used by the frontend or monitoring tools to verify that the API is alive.

Response example:
{
  "status": "ok"
}

3) GET /db-test
Description:
Tests whether the PostgreSQL connection is working.
This is useful for validating the database link before the app starts processing medical data.

Response example on success:
{
  "status": "connected",
  "result": [1]
}

Response example on failure:
{
  "status": "error",
  "message": "Database connection failed: ..."
}

4) POST /chat
Description:
This is the main endpoint for frontend integration.
It accepts a user message, sends it to Groq using the gemma2-9b-it model, and then runs a simple workflow that classifies the interaction and prepares follow-up metadata.

Request body:
{
  "message": "Summarize this patient note"
}

Expected request format:
- The request must be JSON.
- The body must contain a key named "message".
- The message should be a string.

Response structure:
{
  "model": "gemma2-9b-it",
  "response": "...",
  "workflow": {
    "classification": "...",
    "log_status": "Completed",
    "ai_followups": ["..."],
    "should_schedule_followup": false
  }
}

Response field meanings:
- model: the LLM model currently used by the backend.
- response: the answer generated by Groq.
- workflow.classification: the workflow classification for the request.
- workflow.log_status: status of the workflow execution.
- workflow.ai_followups: recommended follow-up actions.
- workflow.should_schedule_followup: boolean flag for follow-up scheduling.

Error response format:
{
  "status": "error",
  "message": "..."
}

Frontend integration guidance
-----------------------------
- Use POST /chat for all AI assistant conversations.
- Send the user's text as a JSON object with a "message" field.
- Show the "response" field in the chat UI.
- Optionally display the workflow values such as follow-ups and classification in a secondary panel or logs.
- If the backend returns an error response, show a friendly message such as "Unable to process your request right now."
- For production, consider adding authentication, request validation, and better error handling.

Example frontend call (JavaScript)
---------------------------------
fetch('http://127.0.0.1:8000/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ message: 'Summarize this patient note' })
})
.then(res => res.json())
.then(data => console.log(data));
